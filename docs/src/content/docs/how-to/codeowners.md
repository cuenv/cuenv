title: CODEOWNERS
description: Generate CODEOWNERS files from .rules.cue
---

cuenv generates CODEOWNERS from `.rules.cue` files. Ownership rules live next to
the directories they describe, and `cuenv sync` aggregates them into one
repository-level CODEOWNERS file.

There is no CODEOWNERS-specific sync subcommand. CODEOWNERS is handled by the
default rules sync provider.

## Quick Start

Create a `.rules.cue` file:

```cue
package cuenv

import "github.com/cuenv/cuenv/schema/rules"

rules.#DirectoryRules & {
    owners: {
        rules: {
            default: {
                pattern: "**"
                owners: ["@myorg/maintainers"]
                order: 0
            }

            rust: {
                pattern: "*.rs"
                owners: ["@myorg/rust"]
                section: "Backend"
                order: 10
            }

            docs: {
                pattern: "docs/**"
                owners: ["@myorg/docs"]
                section: "Documentation"
                order: 20
            }
        }
    }
}
```

Then run:

```bash
cuenv sync
```

For a monorepo with multiple `.rules.cue` files, run workspace sync:

```bash
cuenv sync -A
```

## Commands

Generate or update all generated files, including CODEOWNERS:

```bash
cuenv sync
```

Preview changes:

```bash
cuenv sync --dry-run
```

Validate generated files in CI:

```bash
cuenv sync --check
```

## Directory-Scoped Rules

Each `.rules.cue` file is evaluated independently. Ownership rules are collected
from every discovered `.rules.cue` file and written into the provider-specific
CODEOWNERS path at the repository root.

The generated paths are scoped to the directory that owns the `.rules.cue` file.
For example, this file:

```
services/api/.rules.cue
```

with:

```cue
owners: rules: {
    api: {
        pattern: "*.rs"
        owners: ["@myorg/api"]
    }
}
```

contributes a CODEOWNERS rule for Rust files under `services/api`.

## Rule Fields

| Field | Required | Description |
| --- | --- | --- |
| `pattern` | yes | CODEOWNERS pattern relative to the `.rules.cue` directory |
| `owners` | yes | GitHub/GitLab users, teams, or email addresses |
| `description` | no | Comment emitted above the rule |
| `section` | no | Section heading used to group related rules |
| `order` | no | Deterministic ordering; lower values appear first |

Owner values are schema-validated. Supported forms are:

- `@username`
- `@org/team-name`
- `email@example.com`

## Generated File

cuenv detects the repository provider and writes the conventional CODEOWNERS
path:

| Provider | Output path |
| --- | --- |
| GitHub | `.github/CODEOWNERS` |
| GitLab | `CODEOWNERS` |
| Bitbucket | `CODEOWNERS` |

Generated files include a cuenv header:

```text
# CODEOWNERS file - Generated by cuenv
# Do not edit manually. Configure in .rules.cue and run `cuenv sync -A` to regenerate.
```

Manual edits should be made in `.rules.cue`, then regenerated with `cuenv sync`.
